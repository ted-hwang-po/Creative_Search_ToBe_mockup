# Creative Search ToBe — 기능 리뷰 및 개선 제안

> 작성일: 2026-03-27
> 대상: Creative Search ToBe 프로토타입 (5개 페이지 + v2 데이터셋)
> 분석 방법: 코드/문서 역분석 + 3-Perspective 브레인스토밍 + Pre-mortem

---

## 1. 현재 기능 리뷰

### 1.1 전체 아키텍처 평가

| 항목 | 현재 상태 | 평가 |
|------|-----------|------|
| **No-build 프로토타입** | CDN + Babel Standalone + IIFE 모듈 | 빠른 검증에 적합하나, 페이지당 코드가 비대해지고 있음 (creative-search.jsx가 36K+ 토큰) |
| **Hash 라우팅** | `window.location.hash` 기반 | 5개 페이지에 적합. 그 이상이면 한계 |
| **전역 레지스트리** | `window.__APP.pages/ui/dataset` | 페이지 간 의존성 관리 가능, 네임스페이스 충돌 위험은 낮음 |
| **모사 데이터** | 페이지마다 독립 생성 + v2 공통 데이터셋 병존 | **데이터 이중 관리 문제** — v1과 v2가 혼재 |
| **SVG 썸네일** | data URL 생성 (seed 기반 결정적) | 타입 구분은 가능하나, 실제 소재와의 갭이 큼 |

### 1.2 페이지별 기능 리뷰

#### Page 1: 기존 페이지 (프로모션 소재 검색 결과)
**강점**
- 요약/프로모션/파트너십 3탭 구조가 직관적
- A/B 테스트 세팅까지 한 화면에서 제안하는 것이 차별화
- 브랜드별 소재 유형 비중 차트가 유용

**약점**
- 파일이 36K+ 토큰으로 **단일 파일 비대화** — 유지보수 어려움
- v1 `MOCK_CREATIVES[]` (하드코딩 30여개)와 v2 `creativeAssets[]` (120개 생성형)이 **이중 관리**
- A/B 제안의 근거가 여전히 `abSuggestionForCluster()` 휴리스틱에 의존 (v2 evidence 연동 미완)
- "클러스터" 용어가 코드/변수명에 여전히 남아있음 (UI 텍스트는 변경됨)

#### Page 2: 키 메시지(성분/구성)
**강점**
- 질문형 검색어 UX가 자연스러움
- 성분 vs 구성 두 축 비교가 명확
- 240개 결과의 분포 시각화가 풍부

**약점**
- v2의 `ingredientInsights[]` 연동이 **미완** — 성분 선택 UI(칩/드롭다운)가 아직 없음
- 현재 `focus = ingredient|composition`의 이분법이 **너무 단순** — 실무에서는 더 세분화 필요
- "고효율 예상" 뱃지의 `runDays >= 7` 규칙이 **도메인 타당성 불분명** — 왜 7일인지 근거 없음

#### Page 3: 인플루언서 소재
**강점**
- nano/micro/mid/mega 등급 분류가 실무적
- 인플루언서 60명 × 소재 420개의 데이터 규모가 풍부
- 추천 인플루언서 리스트가 실행 가능한 인사이트 제공

**약점**
- 인플루언서 프로필 사진이 **이모지** — 시각적 신뢰도 저하
- `runDays >= 7`이 인플루언서 소재에서도 동일 기준 → **인플루언서 소재 특성 미반영** (인플루언서 소재는 더 짧은 주기가 일반적)
- 인플루언서-브랜드 매칭 기준이 무작위 배정 → **추천 로직의 설명력 부족**

#### Page 4: 성분/제형/효과 트렌드
**강점**
- 3축(성분/제형/효과) 전환이 Chat으로 가능한 점이 독창적
- 12개 브랜드 × 1200개 소재의 대규모 데이터
- 클러스터 정의(`token:keyMessageType:layoutClass`)가 논리적

**약점**
- **3축 전환 방법이 비직관적** — Chat에서 "제형"을 입력해야 전환되는 것은 발견 가능성(discoverability)이 낮음
- 1200개 소재를 클라이언트에서 한 번에 생성/렌더링 → **초기 로딩 성능 문제 우려**
- skincare/color/device 카테고리 필터가 없음 — 데이터에는 있지만 UI에서 활용 안 됨

#### Page 5: 월간 트렌드
**강점**
- 주차/시즌 브래킷 구조가 실무의 캠페인 사이클과 일치
- 밈 추천 → 자동 스크롤 + 하이라이트 UX가 인상적
- 클러스터당 썸네일 스트립이 시각적으로 풍부

**약점**
- 2026년 1월 **하드코딩** — 월 변경 불가
- `MEME_LIBRARY` 5개 고정 — 확장성 없음
- 밈 추천 흐름이 Chat의 Yes/No 2단계로 제한 — 사용자 맥락 반영 불가

## 2. Pre-mortem 분석

> "이 프로토타입이 실패했다면, 어떤 이유 때문이었을까?"

### 2.1 Tigers (실제 위험 — 높은 확률로 발생)

| # | 위험 | 영향도 | 발생 확률 | 대응 방안 |
|---|------|--------|-----------|-----------|
| T1 | **v1/v2 데이터 이중 관리로 인한 불일치** — 기존 페이지는 자체 `MOCK_CREATIVES`, v2는 `creativeAssets`. 둘이 달라서 시연 시 혼란 | 높음 | 높음 | v2 데이터셋으로 통합 마이그레이션 완료 필요 |
| T2 | **"고효율 예상" 규칙의 도메인 근거 부재** — `runDays >= 7`이 왜 고효율인지 마케터가 납득 못함 | 중간 | 높음 | 실제 성과 데이터 기반 임계값 설정 or 복합 지표(CTR/CVR/ROAS) 도입 |
| T3 | **프로토타입 탈출 난이도** — no-build + IIFE + eval 패턴이 프로덕션 전환 시 전면 재작성 필요 | 높음 | 높음 | 프로토타입의 UI/UX 검증 가치를 명확히 하고, 프로덕션은 별도 스택으로 시작 |
| T4 | **페이지 간 UX 일관성 부족** — 5개 페이지가 각각 독립 발전하여 유사한 정보를 다른 방식으로 표현 | 중간 | 중간 | 공통 패턴 라이브러리 강화 (이미 `patterns.jsx` 존재하지만 활용도 불균일) |
| T5 | **단일 파일 비대화** — `creative-search.jsx`가 이미 36K+ 토큰. 기능 추가 시 유지보수 한계 | 중간 | 높음 | 컴포넌트 분리 (섹션별 파일 분리) |

### 2.2 Paper Tigers (과대 우려 — 실제로는 문제 아님)

| # | 우려 | 왜 문제 아닌가 |
|---|------|----------------|
| PT1 | "SVG 썸네일이 너무 인위적이라 사용자가 이해 못함" | 프로토타입 목적에서는 타입 구분이 핵심이며, SVG로 충분히 구분 가능. 실제 이미지 연동은 프로덕션 단계 |
| PT2 | "5개 페이지가 너무 많아 사용자가 혼란" | 페이지 전환기가 우하단에 있어 비간섭적. 각 페이지가 독립적 질문에 답하므로 동시 사용은 드묾 |
| PT3 | "CDN 의존 때문에 오프라인에서 안 됨" | 프로토타입은 온라인 시연 목적. 오프라인 필요 시 CDN 파일 로컬 캐시 가능 |

### 2.3 Elephants (아무도 말 안 하지만 존재하는 위험)

| # | 위험 | 왜 위험한가 |
|---|------|-------------|
| E1 | **"검색" 기능이 사실상 없음** — 프로토타입의 이름이 "Creative Search"인데 실제 검색 입력 UI가 없음. 고정 쿼리만 있음 | 사용자가 자기 맥락의 검색을 시도하려 할 때 실망. "검색 결과 요약"이라는 프레이밍이 사용자 기대를 만들지만 충족 못함 |
| E2 | **AI Chat이 고정 응답뿐** — Chat 아이콘이 AI 기대를 만들지만, 실제로는 if/else 분기 | 사용자가 자연어 질문을 시도하면 즉시 한계 노출. "AI"라는 라벨을 제거하거나, 한계를 명시해야 |
| E3 | **뷰티 도메인에 과도하게 특화** — 5개 페이지 모두 뷰티/화장품 예시만 사용 | 다른 카테고리(패션, 식품, 테크)로 확장 시 데이터 모델/토큰/카테고리 전면 변경 필요 |
| E4 | **"추천"의 신뢰성 문제** — 모사 데이터의 `predictedScore`가 무작위 생성인데, 이를 기반으로 "추천"을 하면 사용자가 실제 성과 기반이라고 오인할 가능성** | 프로토타입에서는 OK, 하지만 이 기대치로 프로덕션에 진입하면 추천 모델 미비 시 큰 실망 |

### 2.4 분류별 대응 우선순위

| 분류 | 대응 | 시급도 |
|------|------|--------|
| **Launch-blocking** | T1 (데이터 이중 관리), E1 (검색 기능 부재) | 시연 전 해결 필수 |
| **Fast-follow** | T2 (고효율 규칙), T4 (일관성), E2 (Chat 한계 명시) | 1차 시연 후 즉시 |
| **Track** | T3 (프로덕션 전환), T5 (파일 비대화), E3 (도메인 특화), E4 (추천 신뢰성) | 프로덕션 전환 시점 |

## 3. 기능 개선 브레인스토밍 (3-Perspective)

|---|----------|------|--------|--------|

| PM3 | **소재 기획서 자동 생성** | A/B 제안 + 소재 유형 정보를 조합해 "소재 기획 브리프" PDF/마크다운 export | H | M |

|------|----------|------|------|------------|
| 1 | **v2 데이터셋 완전 통합** | E1 | v1/v2 이중 관리(T1)는 시연 전 반드시 해결. 데이터 불일치가 신뢰도를 즉시 훼손 | N (M effort) |

| 7 | **컴포넌트 분리** | E3 | 36K+ 토큰 단일 파일은 이미 읽기/수정 한계. 섹션별 분리 필수 | N (M effort) |

| 9 | **카드 비교 모드** | D2 | A/B 소재 선정은 실무에서 가장 빈번한 행동. 비교 모드가 없으면 스크린샷 + 수작업 | N (M effort) |
| 10 | **URL State 동기화** | E4 | 시연/공유 시 특정 상태를 URL로 전달 가능. 팀 협업에서 큰 차이 | **Y** |

## 8. 공통 컴포넌트 개선

### 8.1 빈 상태 처리 강화
- **현재**: `ContextualEmptyState`가 정의되어 있지만 **실제 사용되는 곳이 거의 없음**
- **개선**: 모든 fold 섹션, 모달, 리스트에 빈 상태 처리 적용

### 8.2 로딩 상태
- **현재**: `LoadingState` 컴포넌트가 있지만, no-build 동기 렌더링이라 사용 안 됨
- **개선**: 대규모 데이터 생성(1200개)을 비동기로 전환할 때 필요. Web Worker 도입과 함께 적용

### 8.3 에러 경계
- **현재**: `boot.js`의 전역 catch만 있음
- **개선**: 페이지 단위 에러 경계(Error Boundary) 추가. 한 페이지 오류가 전체를 다운시키지 않도록

### 8.4 접근성 강화
- **현재**: 기본적인 `aria-*` 적용되어 있으나 불균일
- **개선**:
  - 차트(`MiniBarRow`, `StackedBar`)에 `aria-label` 추가
  - fold 토글에 `aria-expanded` 추가 (일부 누락)
  - 키보드 내비게이션 (Tab/Enter) 검증

## 9. 결론

### 잘 된 점
1. **5개 페이지가 각각 명확한 질문에 답함** — 프로모션, 키메시지, 인플루언서, 트렌드, 월간의 관점 분리가 좋음
2. **공통 UI 패턴이 이미 상당히 발전** — `patterns.jsx`의 컴포넌트들이 재사용 가능
3. **v2 피드백 반영이 체계적** — 용어 변경, 근거 블록, 유형 모달 등의 요구사항이 구조화

### 핵심 개선 방향
1. **데이터 통합** — v1/v2 이중 관리 해소가 최우선
2. **발견 가능성(Discoverability)** — 숨겨진 기능(Chat 전환, 검색)을 표면으로 끌어올리기
3. **분석에서 기획으로의 연결** — "보기만 하는 도구"에서 "다음 행동을 만드는 도구"로 진화
4. **신뢰성 확보** — "고효율 예상", "추천", "근거"가 프로덕션에서도 납득 가능한 수준으로
